# Overkill RE pass27 - pre-level Space-Invaders-like phase

## User clue

Before the scrolling part of each level there is a mini-game / formation phase:
- resembles Space Invaders
- many enemies on empty starfield
- no level tiles
- after that the actual scrolling level begins

This explains why screenshots from "the beginning" of EDRAX do not match LEV1MAP row 0.

## Key conclusion

There is a separate object/state-machine phase before or around the level scroll.
It is not rendered from LEVxMAP/LEVxBLX tiles.

The scrolling tilemap and the pre-level enemy formation are separate systems.

## Important variables

### 0x2356
Current level/resource index.

### 0x2350
Scrolling map counter/pointer.
Known constants:
- 0x0EA0 = 3744 = 13 * 288 full map size
- 0x009C = 156 = 13 * 12 visible-ish rows
- code adds/subtracts 0x0D = 13 to move by one map row

### 0xa7a0
Global/frame counter used by spawn logic.
It is incremented in the per-frame logic:

    incw -0x5860   ; address a7a0

Spawn routines compare it to fixed thresholds:
- 0x32 = 50
- 0x5A = 90
- 0xC8 = 200
- 0xF0 = 240

This strongly looks like timed enemy-wave logic independent of MAP rows.

### 0x2384
Mode/state variable used by object logic.
It is set to 0, 1, 2, 3 in different phases.
Likely connected to player/formation/scrolling or camera mode.

## Formation / Space-Invaders candidate

The strongest candidate is the logic around `0xaa45`.

It spawns a group of 8 objects:

    aa61: mov $0x8,%cx
    loop:
        call 0x6bb6          ; allocate object
        set object fields:
            active = 1
            type/state = 3
            direction/subtype = 0xffff
            positions derived from a8b2/a8b4 and loop counter

This is very formation-like. It creates 8 similar objects at once, which matches a wave/minigame better than tilemap rendering.

## Object spawn gate around aace / aae5

The routine around `aace` uses the global frame counter:

    cmpw $0x32, -0x5860
    cmpw $0x5a, -0x5860

Then it jumps to object-spawn routines.

For level 0, there is a special branch at `aae5`.
For other levels, branches around `ab99/aba3/abad/abb7` select level-specific behavior by 0x2356 and time.

This supports:
- timed enemy waves,
- level-specific formation behavior,
- not directly MAP-driven.

## MAP/tile implication

The editor should treat the level as at least two phases:

1. Pre-level formation phase
   - starfield / no background tiles
   - enemies generated by EXE object logic
   - likely uses 2X2/2X2C/G/SHIP sprite assets

2. Scrolling tilemap phase
   - LEVnMAP + LEVnBLX
   - 13 x 288 cells
   - raw MAP values 1..tile_count are direct background tile indices
   - out-of-range values in some levels are special/unknown map cells

## Practical editor implication

Do not expect "beginning of game level" screenshots to match LEVnMAP row 0.

The GUI should eventually represent:
- Pre-level formation / intro wave
- Main scrolling map

For now, background editor should focus on the scrolling map only.
Enemy/object editor will need separate reverse engineering of object state machine and spawn scripts.

## Useful disassembly anchors

- Main object update loop: 0xa020
- Object dispatch by state: 0xa06e
- Formation-like 8-object spawn: 0xaa45
- Timed spawn gate: 0xaace
- Level-specific object logic: 0xab99..0xabcd
- Spawn position table/pointer: 0xac1b..0xac55
- Map scroll logic: 0x9d91 / 0x9dc4 / 0x9e5e
- End/transition when map reaches 0x009c: check around 0x8dea -> call 0xc948

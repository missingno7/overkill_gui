# Pass59 - enemy storage research, c160 sprite formula

## User goal

Continue toward an enemy overlay that is based on the real runtime object system,
not fake MAP-cell rendering.

## Key user feedback

The c160 screenshot section has 4 enemies using:

    2X2 sprites 147, 148, 149, 150

Decimal to hex:

    147 = 0x93
    148 = 0x94
    149 = 0x95
    150 = 0x96

## New disassembly finding

There is now a much stronger explanation for the sprite IDs.

At 0x5E80..0x5E84:

    inc word [0x233C]
    and word [0x233C], 0x0003

So variable 0x233C is an animation counter cycling:

    0, 1, 2, 3

At 0x8615..0x861B:

    mov ax, [0x233C]
    add ax, 0x0093
    mov [bp+0x08], ax

Therefore this routine selects sprite frames:

    0x93 + 0 = 0x93 = 147
    0x93 + 1 = 0x94 = 148
    0x93 + 2 = 0x95 = 149
    0x93 + 3 = 0x96 = 150

This matches the user-reported c160 enemies exactly.

## Interpretation

This strongly confirms:

    object +0x08 = sprite/frame field

and for this enemy class:

    sprite_frame = 0x93 + animation_counter_0x233C

So the c160 enemy visuals are not stored as static MAP tiles.
They are runtime objects whose sprite frame is computed each tick.

## What is still missing

This does not yet tell us where the 4 enemy spawn positions are stored.

Known unknowns:
- what object behavior/type jumps into the 0x8615 routine,
- what creates the 4 object slots,
- where x/y positions are initialized,
- whether positions are hardcoded in EXE code, a table, or generated by a formation routine,
- what determines the drop.

## Important related routine

Around 0x85CF..0x860A there is nearby logic that:
- computes another sprite field from a table and timer,
- may spawn a child object,
- writes child [object+0x08] = 0x44.

This may be a different enemy class or projectile/child object.
